  1. extract_json_from_stream from adapters/claude_cli.py — this is a parsing

  - Adapters = infrastructure implementations — depends on ports + domain

  application layer does. The issue is just that it has some leaked adapter
  dependencies that should go through ports or be relocated.

● Great question. Yes, the application layer is standard in hexagonal
  architecture — Cosmic Python just introduces it gradually. Here's the fuller
  picture:

  The Four Layers

  Domain    — entities, value objects, business rules (depends on nothing)
  Ports     — interfaces defining how the domain/app talks to the outside
  Application (aka "Service Layer") — use-case orchestration
  Adapters  — infrastructure implementations of ports

  Cosmic Python covers this — they call it the Service Layer (Chapter 4
  onwards). The key insight from the book: the service layer is where you
  orchestrate domain objects + ports to fulfill a use case, but it contains no 
  business logic itself. It's the "glue."

  Your ParsingService.run() is exactly this — it calls the LLM, passes results
  to domain merge logic, saves state via ports. It doesn't decide how to merge
  or validate; it just coordinates.

  The Dependency Rule

  The strict rule (from both hexagonal arch and Cosmic Python) is that
  dependencies point inward:

  CLI → Adapters → Application → Ports → Domain
                        ↓                    ↑
                        └────────────────────┘

  - Domain depends on nothing
  - Ports depend on domain (they reference domain types in their signatures)
  - Application depends on domain + ports
  - Adapters implement ports, so depend on ports + domain
  - CLI/entrypoint wires adapters into the application (depends on everything)

  The CLI is called the Composition Root — the one place where all layers meet
  to wire things together. This is the only place concrete adapter classes
  should appear.

  What Cosmic Python Doesn't Emphasize

  The book is pragmatic Python, so it glosses over a few things that are
  standard in the broader hexagonal/clean architecture literature:

  1. Ports are split into "driving" and "driven" — driving ports are the API
  your app exposes (your service methods), driven ports are what your app needs
  from infrastructure (LLM, VCS, storage). Your ports/ dir is all driven ports,
  which is fine.
  2. The application layer is distinct from domain — some hexagonal
  presentations collapse them, but keeping them separate prevents orchestration
  logic from leaking into entities. Your merge.py is a good example: it
  coordinates domain operations but isn't a domain object itself.
  3. Serialization is a grey area — Cosmic Python uses the Repository pattern to
   hide serialization behind a port. Your tree_to_dict/tree_from_dict straddles
  application and adapter. My proposal puts it in application because the
  service needs it for prompt building, not just persistence.

  The Short Answer

  Your intuition that "everything is domain, port, or adapter" is close — the
  application layer is just the inner ring that orchestrates domain objects 
  through ports. Think of it as the layer that answers "what happens when the
  user runs this command?" without knowing how any infrastructure works.

● Yes — the naming comes from a physical analogy.

  The Analogy

  A port is a socket on a device — like a USB port on your laptop. It defines
  the shape of the connection (what plugs in, what signals it expects) but
  doesn't care what's on the other end.

  An adapter is the cable/dongle that connects a specific external thing to that
   port — like a USB-to-HDMI adapter. It translates between the port's interface
   and whatever the outside world actually provides.

  In Your Codebase

  Port: LLMPort
  class LLMPort(Protocol):
      def invoke(self, prompt, model, allowed_tools, ...) -> LLMResult: ...
  This is the socket. Your application says "I need something that can take a
  prompt and return a result." It doesn't know or care how.

  Adapter: ClaudeCLIAdapter
  class ClaudeCLIAdapter:
      def invoke(self, prompt, model, allowed_tools, ...) -> LLMResult:
          # spawns `claude -p ...` as a subprocess
  This is the dongle that plugs Claude's CLI into that socket. Tomorrow you
  could write an OpenAIAdapter or a FakeLLMAdapter for tests — same port,
  different adapter.

  Another example:

  Port: VCSPort — "I need something that can init a repo and commit."

  Adapter: GitAdapter — "I'll do that with git init and git commit." Could swap
  for a MercurialAdapter or a NoOpVCSAdapter for dry runs.

  The Composition Root (CLI) Plugs Adapters into Ports

  # cli.py — the only place that knows which adapter goes in which port
  llm = ClaudeCLIAdapter()          # plug this adapter...
  vcs = GitAdapter(state_dir)       # ...and this one...
  service = ParsingService(config, llm=llm, vcs=vcs, ...)  # ...into these ports

  The application layer just sees the port shape. The CLI decides what's
  actually plugged in.
