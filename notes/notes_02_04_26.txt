# Haiku Loop Idea
Single phase for now.

Haiku reads large batches, cleans OCR markdown, and annotates document structure inline.

Use comments like:

```markdown
<!-- tree:start id="thm_1_2" type="theorem" anc="ch_1/sec_1_2" title="Theorem 1.2" -->
...
<!-- tree:end id="thm_1_2" -->
```

Attributes:
- id (required, unique)
- title (required)
- type (optional; only for semantic math units)
- anc (optional; advisory only)
- proves (optional; only for type="proof")
- dependencies (optional; list of prerequisite node ids)

Semantic types:
definition, theorem, lemma, proposition, corollary, proof, remark, example, exercise, axiom

Rules:
- Nesting is the source of truth for structure
- Containers (chapters/sections/theorem blocks) should have no semantic type
- Use semantic `type` only for text spans that are actually that thing. E.g. a theorem statement has type="theorem", subsections named after the theorem, further discussions are separate nodes.
- Proofs must be separate nodes with type="proof" and proves="<statement_id>"

proves:
- Only on type="proof"

dependencies:
- Points to earlier nodes required for understanding/justification
- Use appropriately (material prerequisites only, not exhaustive links)
- References existing earlier nodes (forward refs unlikely)

Validator:
Outputs a warning if any of these happen, there may be very rare permissible cases.
- If non proof node has `proves`.
- If proof node does not have `proves` value.
- If `proves` points towards type other than (theorem|lemma|proposition|corollary).
- If `dependencies` points to node if that does note exist.
Outputs if any of are violated:
- Not proper nesting (no crossing spans)
- Unique ids

Example:
raw 
```markdown
# Chapter X
## Theorem A
Preamble
Statement of Theorem A
Remarks
Theorem A Proof
More Remarks
```

Note, this example can have many interpretations. Annotation picks one interpretation.

Interpretation A

ch_x
├── preamble (content=Preamble)
├── thm_a (content=Statement of Theorem A, type="theorem")
├── remarks (content=Remarks, type="remark")
├── thm_a_proof (content=Theorem A Proof, type="proof", proves="thm_a")
└── more_remarks (content=More Remarks, type="remark")

Interpretation B

ch_x
├── preamble (content=Preamble)
├── thm_a (no content)
│   ├── thm_a_statement (content=Statement of Theorem A, type="theorem")
│   ├── remarks (content=Remarks, type="remark")
│   ├── thm_a_proof (content=Theorem A Proof, type="proof", proves="thm_a_statement")
└── more_remarks (content=More Remarks, type="remark")

# Parsing Service

Files are created in the State/ProgresStore dir.

## Basic Loop idea.

l=0
u = l + B

Service writes lines [l,u] of raw.md to raw_1.md

LLM reads raw_1.md and writes clean_1.md

N_1 = number of lines in clean_1.md

clean_1.md will have a syntaxed comment at line c <=N_1. When c < N_1 the LLM has decided to cutoff early where sections split nicely.

N_1 = n_cleaned + 1 (the cutoff comment) + n_unchanged
c = n_cleaned + 1
n_unchanged = N_1 - c

raw.md [u-n_unchanged, u], raw_1.md[u-l-n_changed, u-l], clean_1.md[c+1 N_1] should be identical (lines inclusive denoted here.)

LLM calls an MCP or cli validator (e.g. uv run python validator...) and sees validation passes/warnings/errors.

The validator checks
- clean_1.md post cutoff agrees with raw_1.md
- Whether the inline comments create a consistent tree. This should be like an abstract syntax tree. If syntax is healthy then we should be able to make this into a Node object.

LLM get's opportunity to apply fixes to clean_1.md

The service writes lines [u-n_unchanged, u-n_unchanged + B] of raw.md to raw_2.md.

continue ...
