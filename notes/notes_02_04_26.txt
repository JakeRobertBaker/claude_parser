## Haiku Loop Idea
Single phase for now.

Haiku reads large batches, cleans OCR markdown, and annotates document structure inline.

Use comments like:

```markdown
<!-- tree:start id="thm_1_2" type="theorem" anc="ch_1/sec_1_2" title="Theorem 1.2" -->
...
<!-- tree:end id="thm_1_2" -->
````
Attributes:

* `id`
* `title`
* `type` (optional; only for special semantic/math types)
* `anc` (optional, recommended; advisory only)
* `proves` (optional; warn at validation if a proof node has no `proves`, also warn if `proves` points to anything other than "definition", "theorem", "lemma", "proposition", or "corollary")
* `dependencies` (optional)

Semantic types are: "definition", "theorem", "lemma", "proposition", "corollary", "proof", "remark", "example", "exercise"

Notes

* nesting is the source of truth for structure
* `proves` and `dependencies` are semantic links, not tree structure
* actual statements of definition|...|corollary in the test must be their own type="definition|theorem|lemma|proposition|corollary" nodes. Their proof, if present, is a separate proof type node. Further discussion is a separate node.

Example
raw 
```markdown
# Chapter X
## Theorem A
Preamble
Statement of Theorem A
Remarks
Theorem A Proof
More Remarks
```

Note, this example can have many interpretations. Annotation picks one interpretation.

Interpretation A

ch_x
├── preamble (content=Preamble)
├── thm_a (content=Statement of Theorem A, type="theorem")
├── remarks (content=Remarks, type="remark")
├── thm_a_proof (content=Theorem A Proof, type="proof", proves="thm_a")
└── more_remarks (content=More Remarks, type="remark")

Interpretation B

ch_x
├── preamble (content=Preamble)
├── thm_a (no content)
│   ├── thm_a_statement (content=Statement of Theorem A, type="theorem")
│   ├── remarks (content=Remarks, type="remark")
│   ├── thm_a_proof (content=Theorem A Proof, type="proof", proves="thm_a_statement")
└── more_remarks (content=More Remarks, type="remark")

Inline version of Interpretation B
```
<!-- tree:start id="ch_x" title="Chapter X" -->
# Chapter X

<!-- tree:start id="preamble_1" anc="ch_x" title="Preamble" -->
Preamble
<!-- tree:end id="preamble_1" -->

<!-- tree:start id="thm_a" anc="ch_x" title="Theorem A" -->
## Theorem A

<!-- tree:start id="thm_a_statement" type="theorem" anc="ch_x/thm_a" title="Theorem A" -->
Statement of Theorem A
<!-- tree:end id="thm_a_statement" -->

<!-- tree:start id="thm_a_remarks" type="remark" anc="ch_x/thm_a" title="Remarks" -->
Remarks
<!-- tree:end id="thm_a_remarks" -->

<!-- tree:start id="thm_a_proof" type="proof" anc="ch_x/thm_a" title="Proof of Theorem A" proves="thm_a_statement" -->
Theorem A Proof
<!-- tree:end id="thm_a_proof" -->

<!-- tree:end id="thm_a" -->

<!-- tree:start id="more_remarks" type="remark" anc="ch_x" title="More Remarks" -->
More Remarks
<!-- tree:end id="more_remarks" -->

<!-- tree:end id="ch_x" -->
```

## Parsing Service

l=0
u = l + B

Service writes lines [l,u] of raw.md to raw_1.md

LLM reads raw_1.md and writes clean_1.md

N_1 = number of lines in clean_1.md

clean_1.md will have a syntaxed comment at line c <=N_1. When c < N_1 the LLM has decided to cutoff early where sections split nicely.

N_1 = n_cleaned + 1 (the cutoff comment) + n_unchanged
c = n_cleaned + 1
n_unchanged = N_1 - c

raw.md [u-n_unchanged, u], raw_1.md[u-l-n_changed, u-l], clean_1.md[c+1 N_1] should be identical (lines inclusive denoted here.)

LLM calls an MCP or cli validator (e.g. uv run python validator...) and sees validation passes/warnings/errors.

The validator checks
- clean_1.md post cutoff agrees with raw_1.md
- Whether the inline comments create a consistent tree. This should be like an abstract syntax tree. If syntax is healthy then we should be able to make this into a Node object.

LLM get's opportunity to apply fixes to clean_1.md

The service writes lines [u-n_unchanged, u-n_unchanged + B]
