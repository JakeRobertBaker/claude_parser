Proposed Solution:
I think we need to move to a set of tools, I assume this will be via MCP.
This has already been considered in the past. Currently the StatePort methods are called in ParsingService. However the actions that Haiku does via read and write are coupled with FilesystemStateStore's conventions and file names. The solution to this would have been tools.

Having a tool would also address these cutoff issues. A MCP with read_state, submit_cutoff, write_state, validate_state, commit_state (or something like this - we would need to carefully decide how we do write/validate/commit) would be able to have a read token limit of more than 10k. I assume `FastMCP` is the way to go? 

All Haiku needs to do then is pick the cutoff and submit it's cleaned proposal up to cutoff (no wasted effort on post cutoff alignment).

We also see Haiku read some incorrect file names in the logs. These tools would handle that.

To ensure that Haiku does clean up to it's specified cutoff maybe the validator tool can present the last few lines or tokens of the raw text and the clean text and ask Haiku for final approval.

When we design these tools can we make sure we fully understand how Haiku read and write currently works. When Haiku re writes clean_i.md, e.g. after failed validation, how does it rewrite? Does it cp the old file and make an edit or is it all in context and it just does another write. We will want tools to allow Haiku to work without excess effort issues.

Also, I am concerned that stdout can be unreliable for with claude -p. I think it would be better to have the items of stdout submitted at the end via some kind of commit/submit tool in our mcp.

I think we will need to carefully consider where the wiring to setup this MCP occurs? Do we want this MCP to use the same StatePort that parsing ParsingService uses? Therefore our state is accessed via the same consistent fundamental methods.
